# Staff Guide PWA — Context & Architecture Decisions

> Живой документ. Обновляется по ходу проектирования. Источник — сессия /grill-me 2026-06-23.

---

## Что это

Переиспользуемый PWA-движок для staff guide ресторанов. Построен на базе AlligatorGuide. Движок универсальный — контент и конфигурация заведения вынесены в `data.json`.

---

## Бизнес-модель

- **Вариант 1:** £450 setup + £50/мес (обновления контента делает разработчик через Claude + деплой)
- **Вариант 2:** £700 setup (клиент сам управляет контентом через инструмент — в разработке позже)
- Контент-менеджер сейчас = разработчик. Клиент только пользуется приложением.
- Workflow обновлений: клиент сообщает изменения → разработчик правит `data.json` через Claude → деплой.

---

## Инфраструктура

### Сейчас
- Один репозиторий (`AlligatorGuide`), задеплоен на GitHub Pages
- Только один клиент (Alligator Bar)

### В перспективе (при появлении клиентов)
- Выделенный сервер, купленный домен
- Каждое заведение — свой поддомен (`alligator.staffguide.app`, `ivy.staffguide.app`)
- Движок один, данные разные

### Масштабирование
- Форк папки проекта под каждое заведение
- Движок (`index.html`, `sw.js`) общий — не меняется
- Уникальное на заведение: `data.json`, `images/`, `manifest.json`

## data.json — структура верхнего уровня

```json
{
  "venue": { ... },
  "spiritCategories": { ... },
  "spirits": [ ... ],
  "cocktails": [ ... ],
  "trivia": [ ... ],
  "event": { ... },
  "archived": [ ... ]
}
```

### venue
```json
{
  "name": "The Alligator Bar",
  "logo": "images/logo.svg",
  "accentColor": "oklch(55% 0.18 142)",
  "bgColor": "oklch(20% 0.05 160)",
  "defaultLang": "en"
}
```
Тема применяется через CSS-переменные — не нужно менять код под каждого клиента.

### spiritCategories
Описывает оси вкуса для каждой категории спиритов:
```json
{
  "whisky": {
    "label": { "en": "Whisky", "ru": "Виски" },
    "palateAxes": ["spice", "malt", "fruit", "caramel", "oak"]
  },
  "tequila": {
    "label": { "en": "Tequila", "ru": "Текила" },
    "palateAxes": ["agave", "citrus", "smoke", "sweetness", "pepper"]
  }
}
```

### spirits
Один массив для всех категорий алкоголя:
```json
{
  "id": "bulleit-rye",
  "category": "whisky",
  "name": "Bulleit Rye",
  "palate": { "spice": 4, "malt": 2, "fruit": 3, "caramel": 2, "oak": 3 },
  "addedAt": "2025-09-01",
  "updatedAt": "2025-09-01"
  // ... остальные поля как в текущем WHISKIES[]
}
```

### event
```json
{
  "active": true,
  "title": "London Essence x Alligator",
  "dateStart": "2025-09-15",
  "dateEnd": "2025-09-21",  // опционально; если нет — ивент однодневный
  "items": [
    {
      "id": "london-essence-negroni",
      "name": "L.E. Negroni",
      "description": { "en": "...", "ru": "..." }
      // полная карточка коктейля, не ссылка на cocktails[]
    }
  ]
}
```
- Позиции ивента живут только внутри `event.items` — не дублируются в `cocktails[]`
- Если `active: true` и сегодня между `dateStart` и `dateEnd` → категория перемещается на 1-е место и подсвечивается
- После истечения даты — автоматически уходит в конец навигации, становится неактивной (серая/полупрозрачная)

### archived
Лог удалённых позиций — последние 5 апдейтов:
```json
[
  {
    "menuUpdate": 3,
    "removedAt": "2025-09-01",
    "item": { /* полная карточка позиции */ }
  }
]
```

## Навигация (экраны)

| Порядок | Экран | Условие |
|---|---|---|
| 0 | Main (4-карточный хоум) | Всегда — отдельная кнопка в нав |
| 1 (динамически) | Ивент | Только если активен сегодня; amber dot индикатор вместо мигающего текста |
| 2 | Алкоголь | Всегда — ведёт сразу в выбор типа спирита (не на главный хоум) |
| 3 | Коктейли | Всегда |
| 4 | On the Menu | Всегда |
| 5 | Quiz/Match | Всегда |

**Важное UX-решение (зафиксировано 2026-06-25):** Main и Alcohol — разные кнопки с разным назначением:
- **Main** → 4-карточный хоум (Alcohol / Cocktails / On the Menu / Events)
- **Alcohol** → сразу уровень выбора типа спирита (Whisky / Gin / Rum / …)

### On the Menu
Три подраздела:
1. **Новое** — позиции добавленные в текущем апдейте (`addedAt` = дата последнего деплоя)
2. **Правки** — позиции где изменились цена, описание или другие поля (`updatedAt` = дата последнего деплоя)
3. **86'd (лог удалений)** — внизу экрана, последние 5 апдейтов, кликабельные карточки

Бейдж на иконке "On the Menu" в навигации появляется после каждого апдейта (живёт 14 дней или до следующего апдейта).

---

## Логика "новизны"

- `addedAt` / `updatedAt` проставляется разработчиком при редактировании `data.json`
- Бейдж "NEW" на карточке: видно 14 дней с `addedAt`
- Фильтр "Новинки" в разделе Алкоголь/Коктейли: показывает позиции за последние 14 дней

## Контекстная навигация (activeSpirit)

Глобальная переменная `activeSpirit` — устанавливается при входе в конкретный тип спирита, сбрасывается в `null` при возврате на главный экран.

```
Без контекста (с главного экрана):
Алкоголь → Коктейли (все) → Quiz (все спириты) → [нет симулятора]

С контекстом (например, зашёл в Whisky):
Whisky → Коктейли (только whisky) → Quiz (только whisky) → Match/Sales Sim (только whisky)
```

- Свайп влево всегда идёт по этой цепочке
- Контекст сохраняется даже если внутри Whisky перешёл в Bourbon — свайп всё равно покажет whisky-коктейли
- Match (гость-сомелье) = Sales Simulator. Один и тот же экран, данные фильтруются по `activeSpirit`

## Коктейли

```json
{
  "id": "negroni",
  "name": "Negroni",
  "baseSpirit": "gin",
  "signature": false,
  ...
}
```

- `baseSpirit` — один, всегда главный компонент (gin для Negroni, даже если есть вермут)
- `signature: true` — авторский рецепт бара, не классика
- Фильтры на экране Коктейли генерируются из ключей `spiritCategories` автоматически

## ADR (Architecture Decision Records)

### ADR-1: Один файл на заведение
**Решение:** Каждое заведение имеет свой `data.json`, а не общую мастер-базу.
**Причина:** Изолированность деплоев, нет риска показать чужое меню, проще управлять.

### ADR-2: Один массив `spirits[]` + `spiritCategories`
**Решение:** Все спириты в одном массиве, схема осей вкуса — в отдельном объекте категорий.
**Причина:** Движок рендерит любую категорию без изменений кода. Масштабируется на ром, джин и т.д.

### ADR-3: Брендинг через CSS-переменные из `data.json`
**Решение:** `venue.accentColor` и `venue.bgColor` применяются как CSS-переменные при загрузке.
**Причина:** Не нужно трогать код под каждого клиента.

### ADR-4: Поддомены для мультиарендности (отложено)
**Решение:** В перспективе — поддомены. Сейчас — один репо для Alligator.
**Причина:** Нет смысла строить инфраструктуру до появления второго клиента.

### ADR-5: Навигация по алкоголю — умная иерархия
**Решение:** Глубина навигации зависит от категории. Whisky → подкатегории → карточки с фильтром по региону. Tequila/Vodka/Cognac → сразу карточки. Логика Scotch (открывается на All Scotland + пиллы регионов) сохраняется без изменений.
```
Алкоголь
├── Whisky → Bourbon/Scotch/Rye… → карточки (+ регион-фильтр для Scotch)
├── Tequila → карточки
├── Cognac  → карточки
└── Vodka   → карточки
```
`spiritCategories` содержит поле `levels` — движок знает сколько уровней показывать.
**Причина:** Сохраняем привычный UX, не ломаем то что уже работает хорошо.

### ADR-6: Ивент-позиции изолированы
**Решение:** Коктейли ивента хранятся только в `event.items`, не дублируются в `cocktails[]`.
**Причина:** Временные позиции не должны засорять основной каталог.

### ADR-6: Управление контентом через разработчика
**Решение:** Разработчик правит `data.json` через Claude и делает деплой. Клиент не трогает данные.
**Причина:** Бизнес-модель £50/мес за обновления. Инструмент самообслуживания — отдельная задача.
